# Watchdog、Brownout 与 GPIO 冲突案例

用途：帮助 Embex 分析运行期重启、供电异常和引脚冲突。

## Watchdog reset

常见原因：

- `setup()` 中外设初始化卡死。
- `loop()` 中长时间阻塞。
- 死循环等待传感器响应。
- 外设库在错误接线下阻塞。

诊断方式：

- 检查串口是否反复输出 `[BOOT]`。
- 在每个初始化步骤前后输出日志，定位卡在哪一步。
- 将复杂外设初始化拆开，逐个验证。
- 优先写最小可运行固件，再逐步加外设。

## Brownout

常见原因：

- USB 供电不足。
- 蜂鸣器、OLED、传感器同时工作导致电压跌落。
- 面包板接触不良。
- 线太长或电源质量差。

诊断方式：

- 查找串口中的 `brownout`。
- 先断开大负载外设。
- 用稳定 3.3V 供电。
- 蜂鸣器等负载使用三极管或 MOS 管驱动。

## GPIO 冲突与接线错误

常见场景：

- 同一个 GPIO 同时分配给 OLED CLK 和 I2C SCL。
- 外设接到启动相关 GPIO，并且外设电平影响启动模式。
- SPI OLED 的 SCL/SDA 被误认为 I2C SCL/SDA。
- ESP32 原版 GPIO6-GPIO11 被误用于外设。
- 杜邦线接错、相邻 GPIO 混接、VCC/GND/DATA 顺序错误。

## 项目案例修正

本项目曾观察到“接入 DHT11 后烧录异常，断开相关接线后恢复”的现象。后续确认真实原因是当时接线错误，不是 GPIO11 或 DHT11 本身问题。

因此 Embex 后续诊断必须这样表述：

- 这是“误接线影响烧录/下载”的历史案例。
- 不能把 GPIO11 直接判定为不可用。
- 不能在当前烧录成功时继续报告该故障。
- 当出现烧录失败时，可以建议检查最近新接的外设、DHT11 DATA 线、供电、上拉电阻和实际 GPIO 编号。
