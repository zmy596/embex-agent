# Embex ReAct 闭环开发流程

用途：指导 Embex 将自然语言任务转化为可执行的 ESP 嵌入式开发闭环。

## 标准流程

1. 理解任务：判断用户是在询问信息、配置硬件、生成固件、执行烧录，还是诊断错误。
2. 读取上下文：加载硬件配置、已启用外设、板卡引脚表、长期记忆和相关 RAG 知识。
3. 形成计划：明确本轮需要哪些工具、哪些输入、验收标准是什么。
4. 生成源码：由模型生成完整 Arduino `main.cpp`；辅助库只作为可选函数集合。
5. 创建工程：生成 PlatformIO 工程，包括 `platformio.ini`、`src/main.cpp`、必要库依赖。
6. 编译：运行 PlatformIO build，解析错误和资源占用。
7. 烧录：有串口时上传固件；没有串口时只完成编译并提示缺失条件。
8. 串口监控：采集真实运行日志，作为诊断和验收证据。
9. 结果判断：根据任务指定的验收规则判断成功、失败或需要用户观察确认。
10. 迭代修复：失败时根据日志和硬件配置调整代码、引脚、库或接线建议。

## ReAct 约束

- Thought：分析任务和约束，但最终页面应展示精炼可追踪结论。
- Action：调用明确工具，例如 RAG 检索、Skill、MCP、编译烧录工具。
- Observation：保留工具返回、编译日志、烧录日志、串口日志。
- Reflection：用观察结果判断任务是否达到验收标准。

## 工具职责

- RAG：提供规则、板卡资料、外设资料和调试经验。
- Skill：提供领域能力，例如引脚分析、外设组合项目规划。
- MCP：读取文件、项目结构、Git 状态、串口硬件状态。
- PlatformIO 工具：执行工程生成、编译、烧录、串口监控。

## 失败节点定位

每轮失败必须说明失败节点：

- 任务理解失败。
- 知识检索失败。
- main.cpp 校验失败。
- 编译失败。
- 烧录失败。
- 串口监控失败。
- 硬件行为未达到验收标准。
- 模型总结失败。

失败结论不能只说“工具后端故障”或“稍后重试”，必须提供具体证据。
